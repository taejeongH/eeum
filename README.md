# 공통 프로젝트 기획서

## 1. 혼자 사는 사람을 위한 생활 패턴 이상 감지 서비스

### 1. 문제 정의

1인 거주자는 낙상·무응답과 같은 위험 상황이 발생해도

즉시 발견되기 어렵다.

또한 가족이 존재하더라도 **누가 언제 확인했는지,
추가 방문이 필요한지에 대한 정보가 공유되지 않아
돌봄이 단절되는 문제**가 있다.

---

### 2. 서비스 개요

본 서비스는

**AIoT 위험 감지**와 **가족 공유 다이어리**를 결합한 시스템이다.

- AI는 위험 상황을 **이벤트 단위로 감지**
- 가족은 위험 이후의 **대응을 기록·공유**

# 협업 규칙 가이드

본 문서는 프로젝트 협업 시 **이슈 관리, 브랜치 전략, PR 규칙, 커밋 컨벤션**을 통일하기 위한 가이드입니다.

---

## 1. 이슈 & PR 템플릿

### ✅ PR 유형

> 어떤 변경 사항이 있었나요?

* [ ] 새로운 기능 추가
* [ ] 버그 수정
* [ ] 코드에 영향을 주지 않는 변경사항 (오타 수정, 탭 사이즈 변경, 변수명 변경)
* [ ] 코드 리팩토링
* [ ] 테스트 코드 추가
* [ ] 주석 추가 및 수정
* [ ] 문서 수정
* [ ] 빌드 부분 혹은 패키지 매니저 수정
* [ ] 파일 혹은 폴더명 수정
* [ ] 파일 혹은 폴더 삭제

---

### 🚀 작업 내용

> 이번 PR에서 작업한 내용을 구체적으로 설명해주세요. (이미지 첨부 가능)

* 작업한 내용 1
* 작업한 내용 2

### 💬 기타 사항 or 추가 코멘트

## 2. 브랜치 규칙

* **이슈 생성 후 자동으로 브랜치 생성**
* 브랜치명 규칙:

```
[이슈번호]-[타입]-[간단한 설명]
```

* 예시:

```
15-feat-user
```

## 4. 커밋 규칙 (Commit Convention)

* **모든 커밋 메시지는 소문자로 작성**

### 커밋 유형

| 타입               | 의미                                    |
| ---------------- | ------------------------------------- |
| feat             | 새로운 기능 추가                             |
| fix              | 버그 수정                                 |
| docs             | 문서 수정                                 |
| style            | 코드 formatting, 세미콜론 누락 등 코드 변경이 없는 경우 |
| refactor         | 코드 리팩토링                               |
| test             | 테스트 코드, 테스트 리팩토링                      |
| chore            | 패키지 매니저 수정, 기타 설정 변경 (.gitignore 등)   |
| design           | css 등 사용자 ui 디자인 변경                   |
| comment          | 필요한 주석 추가 및 수정                        |
| rename           | 파일 또는 폴더명 수정 또는 이동                    |
| remove           | 파일 삭제 작업만 수행                          |
| !breaking change | 큰 api 변경                              |
| !hotfix          | 긴급하고 치명적인 버그 수정                       |

---

### 커밋 메시지 예시

```
feat: 게시판 기능 개발 (#32)
```

---

✅ 위 규칙을 통해 **일관된 협업과 코드 품질 유지**를 목표로 합니다.
