## 🟢 이슈(Issue) 규칙
모든 작업은 이슈 생성부터 시작합니다. 이슈 템플릿을 준수하여 작성해 주세요.

### 1. 이슈 생성 템플릿
- **Title:** `깃모지 [유형] : Issue 제목` (예: ✨ [feat] : 실시간 스켈레톤 추출 기능 구현)
- **Labels:** 해당 파트 표시 (`FE`, `BE`, `AI`, `IoT`)
- **Assignees:** 작업 담당자 지정

### 2. 작성 내용
- **Description:** 구현할 구체적인 내용을 번호와 함께 기술합니다.
- **Todo:** 작업을 완료하기 위해 필요한 세부 체크리스트를 작성합니다.

## 🌿 브랜치(Branch) 규칙
이슈를 생성하면 자동으로 생성되는 브랜치 기능을 활용하거나, 아래 규칙에 따라 브랜치를 생성합니다.

### 1. 주요 브랜치
- main: 제품 배포를 위한 가장 안정적인 브랜치입니다. (Production)
- develop: 다음 출시 버전을 위해 기능을 통합하는 브랜치입니다. 모든 개발의 기준점이 됩니다. (Staging/Integration)

### 2. 작업 브랜치 (Feature Branch)
- 새로운 기능 추가나 버그 수정은 반드시 develop 브랜치에서 분기한 개별 브랜치에서 진행합니다.
- 브랜치 생성 위치: develop
- 브랜치명 형식: [이슈번호]-[유형]-[작업자]
- 예시: 15-feat-user

### 3. 브랜치 흐름
- develop ➔ feature ➔ develop (PR/Review) ➔ main (Release)

## 💬 커밋(Commit) 컨벤션
커밋 메시지는 한눈에 파악할 수 있도록 아래 규칙을 따릅니다.

### 1. 커밋 유형 (영어 소문자)
| 유형 | 의미 |
| :--- | :--- |
| **feat** | 새로운 기능 추가 |
| **fix** | 버그 수정 |
| **docs** | 문서 수정 |
| **style** | 코드 formatting, 세미콜론 누락 등 코드 변경이 없는 경우 |
| **refactor** | 코드 리팩토링 |
| **test** | 테스트 코드 추가 및 수정 |
| **chore** | 빌드 부분 혹은 패키지 매니저 수정 |
| **design** | CSS 등 UI 디자인 변경 |
| **comment** | 필요한 주석 추가 및 변경 |
| **rename** | 파일 혹은 폴더명 수정/이동 |
| **remove** | 파일 삭제 작업만 수행 |
| **!BREAKING CHANGE** | 커다란 API 변경 |
| **!HOTFIX** | 급하게 치명적인 버그를 고쳐야 하는 경우 |

### 2. 커밋 메시지 형식
> `유형: 요약 내용 (#이슈번호)`

**예시:** `feat: 게시판 기능 개발 (#32)`

## 🚀 PR(Pull Request) 규칙
코드 리뷰를 거쳐 `main` 브랜치에 반영하기 위해 아래 절차를 따릅니다.

### 1. PR 유형 체크
- [ ] 새로운 기능 추가
- [ ] 버그 수정
- [ ] 코드에 영향을 주지 않는 변경사항 (오타, 변수명 변경 등)
- [ ] 코드 리팩토링
- [ ] 테스트코드 추가
- [ ] 주석/문서 수정
- [ ] 빌드/패키지 매니저 수정
- [ ] 파일/폴더명 수정 및 삭제

### 2. PR 내용 작성
- **작업 내용:** 이번 PR에서 작업한 내용을 구체적으로 설명합니다. (필요 시 이미지 첨부)
- **기타 사항:** 리뷰어가 참고할 코멘트나 관련 참고 자료를 기록합니다.

---

## 🛠 코드 리뷰 및 승인
- 모든 PR은 최소 **1명 이상의 승인**이 있어야 머지할 수 있습니다.
- 머지 방식은 커밋 히스토리 관리를 위해 **Squash and Merge**를 기본으로 합니다.
